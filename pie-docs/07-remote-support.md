# 원격지원

## 목표

고객의 명시적 동의와 조직 정책 아래 채팅, 터미널, 데스크톱을 제공하고 지원 작업을 티켓,
자산, 코드 변경, 공수, 감사 기록과 연결한다.

## 진입 방식

- 서비스 티켓에서 세션 생성
- 고객이 앱에서 지원 요청
- 엔지니어가 고객 자산에서 세션 요청
- `pie://support/<invite>` 딥링크
- 만료되는 초대 코드
- 무인 접근 정책에 따른 예약 세션

## 세션 참여자

- 고객 사용자
- 주 지원 엔지니어
- 추가 전문가
- 세션 관리자
- 보기 전용 감사자
- 제한된 게스트

## 세션 상태

`요청 → 고객 동의 대기 → 연결 중 → 활성 → 일시 중지 → 종료 → 검토 완료`

동의가 철회되거나 정책이 만료되면 입력을 즉시 차단하고 연결을 종료한다.

## 권한 등급

| 등급 | 허용 범위 |
|---|---|
| 관전 | 화면과 터미널 출력 보기 |
| 채팅 | 관전과 세션 채팅 |
| 터미널 조작 | 지정 터미널 입력 |
| 데스크톱 조작 | 마우스와 키보드 입력 |
| 파일 전송 | 명시된 방향의 파일 송수신 |
| 관리자 | 참여자와 조작권, 녹화, 세션 종료 관리 |

권한은 세션 중에도 회수할 수 있다. 한 명만 조작 가능한 자원은 현재 드라이버를 명시한다.

## 원격 터미널

- PTY 생성과 셸 선택
- 시작 작업 디렉터리 제한
- 로컬·WSL·SSH 대상
- UTF-8, 한글 IME, 터미널 크기 동기화
- 관전과 단일 드라이버
- 조작권 전달·회수
- 명령 승인과 위험 명령 차단
- 재연결과 스크롤백 복원
- 출력 스트리밍과 backpressure
- 명령·출력 녹화와 재생

기존 `cli-relay`의 room, participant, view/control, driver와 reconnect 개념을 출발점으로 사용하되
현재 shared HS256 secret, subject-only pairing과 query token 계약을 production에 그대로 사용하지
않는다. 초대, 사용자, 세션, 동의, 감사와 이력은 중앙 Control Plane에 영속화하고 Orca의 host proof,
device binding, scoped token과 E2EE 구현을 security baseline으로 삼는다. 상세 결정은
[`ADR-0011`](../docs/adr/0011-self-hosted-platform-dependencies.md)을 따른다.

## 원격 데스크톱

- 화면 보기와 조작
- 다중 모니터
- 해상도와 품질 조절
- 대역폭 적응
- 클립보드 정책
- 파일 전송
- 화면 캡처와 녹화
- 고객 입력 우선권
- 관리자 권한 전환 안내
- 원격 재부팅 후 재연결
- 여러 전문가 참여

초기에는 검증된 원격 데스크톱 엔진을 사이드카로 통합하고, Relay와 권한·감사 계층을 Pie가
소유한다.

## 플랫폼 권한과 승격

| 플랫폼 | 필수 고려사항 |
|---|---|
| Windows | UAC, Secure Desktop, 서비스 세션, 사용자 전환, 재부팅 후 재연결 |
| macOS | 화면 기록, 손쉬운 사용, 마이크·카메라 권한, TCC 변경 안내 |
| Linux | X11·Wayland 지원 차이, 데스크톱 세션, polkit, 배포판별 패키지 |

- 권한이 부족하면 연결 실패와 관전 전용 상태를 구분한다.
- 관리자 승격 전 고객에게 대상 프로세스와 예상 영향을 다시 고지한다.
- 고객이 로컬 입력이나 긴급 중지 명령으로 원격조작을 즉시 차단할 수 있어야 한다.
- 무인 접근은 Agent 설치, 장비 관리자 승인, 정책 만료, 제거 절차를 별도로 요구한다.
- 재부팅과 사용자 전환 중 기존 capability를 재사용하지 않고 새 상태를 검증한다.
- 플랫폼이 지원하지 않는 기능은 안전하게 비활성화하고 우회 입력을 시도하지 않는다.

## Agent와 Relay 신원

- Edge Agent 설치 시 장비별 키와 인증서를 발급한다.
- 인증서 갱신·폐기, 장비 복제와 키 유출 탐지 절차를 제공한다.
- Relay는 네트워크 위치가 아니라 사용자·Agent·세션 capability를 모두 검증한다.
- Agent가 지원하는 화면, 터미널, 파일, 승격 capability를 handshake로 보고한다.
- 구버전 Agent와 Relay의 최소 지원 버전과 강제 업데이트 정책을 둔다.
- 리전 선택, 경로 전환, Relay 장애 중 재연결은 세션 동의와 권한을 보존한다.
- Agent 업데이트 패키지의 서명과 rollback 방지를 검증한다.

## 파일 전송

- 업로드·다운로드 방향별 허용
- 파일 선택과 대상 경로 확인
- 용량·확장자·악성코드 정책
- 해시와 전송 결과
- 고객 확인
- 다운로드와 반출 감사
- 티켓 첨부 또는 격리 저장

## 세션 기록

- 참여자 입장·퇴장
- 권한 변경과 조작권
- 고객 동의와 철회
- 터미널 입력·출력
- 화면 녹화 정책과 동의
- 파일 전송
- 실행한 Runbook과 결과
- 연결 품질과 중단
- 작업시간
- 엔지니어·AI 요약

## 보안 원칙

- 고객 장비는 inbound 포트를 열지 않고 outbound 연결을 사용한다.
- 초대와 세션 토큰은 단기·대상 제한·폐기 가능해야 한다.
- Relay는 신원을 검증하고 발신자 필드를 강제한다.
- Edge Agent와 Relay 서비스 신원을 사용자 계정과 분리한다.
- 화면에서 권한을 숨기는 것만으로 입력을 차단하지 않는다.
- 녹화와 민감정보 수집은 정책과 고객 동의를 명시한다.
- 세션 종료 후 제어 토큰과 임시 파일을 폐기한다.

## P1 완료 기준

티켓에서 원격 터미널 세션을 만들고 고객이 같은 Electron 앱의 제한 모드로 동의·참여하며,
엔지니어가 조작권을 받아 작업한 뒤 세션 기록·요약·공수가 티켓에 남아야 한다.
지원 대상 플랫폼에서는 권한 부족, 동의 철회, Agent 폐기 후 입력과 재연결이 즉시 차단되어야 한다.
