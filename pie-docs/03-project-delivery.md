# 프로젝트 수행

Team, Initiative, Cycle, Intake, SavedView와 ProjectUpdate를 포함한 빠른 프로젝트 실행 작업면은
[프로젝트 실행 모델](./27-project-execution-model.md)을 따른다. 이 문서는 SI 납품, 승인, 일정과
산출물 규칙을 추가로 정의한다.

## 목표

SI 프로젝트의 계약 범위를 일정, 인력, 요구사항, 코드, 산출물, 검수, 비용과 연결한다.
단순 칸반이 아니라 변경과 검수의 근거를 보존하는 수행 시스템을 제공한다.

- 프로젝트 개요

- 마일스톤과 일정
- 요구사항

- 인력과 공수

- 고객 검수
- 프로젝트 손익

## 프로젝트 기본정보

- 고객사, 계약, 프로젝트 유형
- 기간, 기준선, 상태, 건강도
- PM, PL, 팀, 고객 담당자
- 저장소, 배포환경, 서비스, 자산

## WBS와 일정

- 작업과 하위 작업
- 담당자, 검토자, 승인자
- 계획 시작·종료, 실제 시작·종료
- 선후행 관계와 차단 사유
- 마일스톤과 기준선
- 목록, 칸반, 간트, 캘린더 보기
- 반복 작업과 템플릿
- 지연, 과부하, 주요 경로 경고
- 작업에서 Workspace, 티켓, 문서 열기

## 업무 실행과 칸반

- 중앙 `WorkItem` 칸반과 로컬 Workspace Board를 별도 보기로 제공
- 하나의 업무에 여러 Workspace, Worktree, Agent Session, PR·MR 연결
- 업무에서 저장소·host·agent를 선택해 `Workspace에서 열기`
- 앱 밖에서 시작한 AI 세션을 처리하는 Intake source
- 업무 설명, 완료 조건, 의존성, 담당자, 검토자, 승인자
- AI session timeline, 변경 파일, commit, test, build, review와 산출물
- 세션 연결 수정과 외부 issue mapping의 이력·감사
- agent run 상태와 WorkItem 상태, Workflow 승인 상태의 분리

세션 시작, PR 생성, 테스트 성공은 상태 변경의 근거 후보다. 에이전트가 완료했다고 응답하거나
Workspace 열이 이동했다는 이유만으로 프로젝트 업무를 완료하지 않는다. 프로젝트 정책이 허용한
전이는 완료 조건과 최신 version을 다시 검사한 뒤 적용한다.

- 계약 범위 포함 여부

- 리스크와 대안
- 내부 검토, 고객 승인, 반려
- 승인 전 관련 작업 실행 제한
- 변경 계약과 청구 항목 연결
- 개발·테스트·배포·검수 결과 연결

- 역할과 기술별 인력 배정

- 직원 원가와 고객 청구 단가

- 예상 잔여 공수와 완료 시점

- 의사결정 내용과 근거
- 회의·문서·변경요청 연결
- 기한과 후속 작업
- AI 위험 요약과 추세

## 산출물과 검수

- 산출물 폴더와 템플릿
- 제출 버전과 파일 해시
- 내부 검토, 고객 제출, 반려, 재제출
- 검수 항목과 통과 조건
- 테스트 결과와 릴리스 연결
- 고객 확인과 전자서명
- 납품 패키지와 종료 보고서

## 테스트와 품질

- 테스트 계획과 케이스
- 요구사항 연결
- 실행 결과와 증빙
- 결함, 심각도, 재현 절차
- 수정 커밋과 재시험
- UAT와 고객 확인
- 결함 재오픈률과 누출 지표

## 상태 흐름

`제안 → 준비 → 수행 → 고객 검수 → 안정화 → 종료 → 유지보수`

단계 전환은 필수 산출물, 승인, 미해결 위험, 청구 상태를 검사한다. 강제 전환은 사유와
승인자를 감사 로그에 남긴다.

## P2 완료 기준

계약에서 생성된 프로젝트가 WBS, 요구사항, 변경요청, 공수, 산출물, 검수, Workspace와
연결되고, 계획 대비 일정·공수·비용을 조회할 수 있어야 한다.

AI 실행과 업무 연결의 세부 계약은 [AI 작업 프로젝트 포털](./19-ai-project-portal.md), 상태·동기화
실패 시나리오는 [AI 프로젝트 포털 구현 위험](./20-ai-project-portal-risk-register.md)을 따른다.
