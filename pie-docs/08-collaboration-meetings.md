# 협업과 회의

## 목표

대화를 별도 메신저에 고립시키지 않고 고객, 프로젝트, 티켓, 원격 세션, 의사결정과 연결한다.
범용 커뮤니티 메신저보다 업무 문맥과 기록을 우선한다.

회의·회의록의 현재 구현 기준, 상세 데이터 모델과 순차 backlog는
[회의·회의록 제품 로드맵](./35-meeting-product-roadmap.md)을 따른다.
채팅은 Slack 전체 기능 복제가 아니라 [채팅 핵심 제품 로드맵](./36-chat-core-roadmap.md)의
P0 일상 대화 코어와 P1 Pie 업무 연결 범위로 제한한다.

## 대화 공간

- 조직 공지 채널
- 팀 채널
- 고객사 채널
- 프로젝트 채널
- 티켓 대화
- 원격 세션 사이드 채팅
- 1:1 메시지
- 회의 채팅

## 메시지 기능

- 텍스트, Markdown, 코드
- 파일과 이미지
- 답글과 스레드
- 멘션과 반응
- 읽음·안 읽음
- 고정 메시지
- 수정·삭제 이력
- 전문 검색
- 보존 기간
- 메시지에서 작업·티켓·결정 생성
- AI 요약과 액션 아이템

고객 공개 메시지와 내부 메시지를 명확하게 구분한다. 같은 티켓에서 작성하더라도 내부 메모는
고객 계정과 게스트에게 전송되지 않는다.

## 화상회의

- 프로젝트·티켓·원격 세션에서 회의 생성
- 음성·영상
- 화면과 특정 창 공유
- 다자간 회의
- 대기실과 호스트 승인
- 발표자와 참여자 권한
- 장치 선택과 연결 진단
- 채팅과 파일 공유
- 녹화 동의와 상태 표시
- 녹화, 전사, 자막
- 회의 종료 후 AI 요약
- 결정사항과 후속 작업 생성

미디어는 일반 업무 WebSocket이나 터미널 Relay에 섞지 않고 전용 SFU를 사용한다. Pie는
회의 권한, 업무 문맥, 녹화 메타데이터, 보존정책을 관리하고 `meeting` 배포 profile의 Self-hosted
LiveKit에 room·participant가 제한된 단기 token을 발급한다. LiveKit 장애나 profile 비활성화는 회의
capability만 제한하며 Core 프로젝트·대화 API를 중단시키지 않는다. 상세 경계는
[`ADR-0011`](../docs/adr/0011-self-hosted-platform-dependencies.md)을 따른다.

## 일정

- 프로젝트와 고객 회의
- 정기점검과 변경 작업
- 내부·고객 참석자
- 시간대 처리
- 캘린더 연동
- 회의 링크와 관련 자료
- 종료 후 회의록과 참석 기록

## 알림

- 앱 내 알림함
- OS 데스크톱 알림
- 멘션, 배정, 승인 요청
- 회의 시작
- SLA 위험
- 원격지원 요청
- 에이전트·자동화 완료
- 사용자별 채널과 방해 금지 시간
- 이메일·메신저 연동

## 권한 규칙

- 채널 멤버십은 고객사와 프로젝트 권한을 상속할 수 있다.
- 외부 참가자는 초대된 공간과 회의만 접근한다.
- 녹화·전사는 시작 전 고지하고 정책에 따라 동의를 받는다.
- 삭제된 메시지의 감사 메타데이터와 실제 본문 보존은 정책을 분리한다.

## 현재 회의 구현 기준

2026-07-21 기준으로 회의 lifecycle, 초대된 참여자의 LiveKit room 참여, 다자간 음성·영상,
갤러리·발언자 보기, 카메라·마이크·스피커 선택과 미리보기, 화면·창 공유, capture 유형별 동의,
MP4 녹화·재생, 실시간 자막, 시간·화자별 사후 전사와 수정 이력, 근거가 연결된 AI 회의록,
결정·액션 아이템 검토와 WorkItem 전환, 일정·반복·시간대·알림, 내부 사용자 검색, guest link,
Google·Microsoft calendar outbound adapter, 종료 후 Recap, 보존·자동 삭제·legal hold와 감사되는
JSON 내보내기 수직이 존재한다.

실사용 완결을 위해 실장비·교차 플랫폼 media E2E, guest 초대를 소비하는 Desktop 흐름,
calendar OAuth lifecycle, 부분 redaction·관리자 step-up·backup 복원 후 재삭제가 남아 있다. 회의록
생산성은 보류했으며 완료 판정과 나중의 재개 기준은 상세 로드맵을 따른다.

## P2 완료 기준

프로젝트와 티켓에서 내부·고객 대화를 구분해 지속적으로 주고받고, 회의를 시작해 화면을
공유하며, 종료 후 요약·결정·후속 작업이 원래 문맥에 기록되어야 한다.
