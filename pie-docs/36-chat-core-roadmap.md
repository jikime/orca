# 채팅 핵심 제품 로드맵

## 목적

Pie 채팅은 Slack의 모든 기능을 복제하지 않는다. 첫 목표는 팀이 다른 메신저로 이동하지 않고
업무 대화를 이어 갈 수 있는 **채널 중심의 신뢰 가능한 채팅**이다. 회의와 회의록은
[회의·회의록 제품 로드맵](./35-meeting-product-roadmap.md)이 별도로 소유하고, 이 문서는 회의 전후의
비동기 대화와 회의 중 텍스트 대화에 공통으로 재사용할 채팅 코어만 다룬다.

범위 판단은 Slack 공식 문서의 다음 사용자 흐름을 참고했다.

- [채널](https://slack.com/help/articles/360017938993-What-is-a-channel): 주제별 대화 공간과 스레드
- [DM](https://slack.com/help/articles/212281468-Understand-direct-messages): 1:1·소규모 그룹 대화
- [검색](https://slack.com/help/articles/202528808-Search-in-Slack): 대화 기록 탐색
- [메시지 수정·삭제](https://slack.com/help/articles/202395258-Edit-or-delete-messages): 작성자 메시지 관리
- [대화별 알림](https://slack.com/help/articles/360056534254-Manage-notifications-for-specific-channels-and-direct-messages): 알림과 뮤트

이는 UX 원칙 참고이며 Slack API·화면·데이터 모델 호환을 뜻하지 않는다.

## 이번에 닫는 범위

### P0 · 일상 대화 코어

- 조직 채널 생성·목록과 권한이 있는 관리자의 멤버 추가
- 1:1 DM과 그룹 DM
- 실시간 메시지 송수신과 최신 메시지 진입
- Markdown·코드·링크·목록·이모지 작성과 안전한 렌더링
- 이미지·파일 첨부와 권한이 적용된 다운로드
- 스레드 답글과 답글 수
- 빠른 이모지 반응
- 사용자 멘션, `@channel`, `@here`
- 읽음 커서, 채널별 안 읽음 수, 멘션 알림함
- 사용자 접속 상태와 입력 중 표시
- 메시지 수정·삭제와 `(edited)`·삭제 tombstone
- 메시지 고정과 고정 목록
- 권한 인식 전문 검색과 결과 위치 이동
- 채널·DM 뮤트

### P1 · Pie 업무 연결

- 메시지에서 WorkItem 생성과 원문 역링크
- 프로젝트·티켓·회의가 소유한 문맥 채널
- 회의 상세 화면과 같은 채팅 코어를 재사용하는 회의 채팅
- OS 데스크톱 알림과 사용자별 알림 설정
- 채널 멤버 제거, 채널 주제·설명, 보관

P1은 Slack 모방이 아니라 Pie의 업무 문맥을 완성하는 범위다. P0의 메시징 신뢰성을 먼저 닫은 뒤
구현한다.

### 이번 범위에서 제외

- Slack Connect와 외부 조직 간 공유 채널
- 앱 마켓플레이스, 봇 플랫폼, Workflow Builder
- Canvas, Lists, Clips와 이메일 수신 주소
- 사용자 정의 이모지와 전체 이모지 카탈로그
- 예약 전송, 나에게 알림, 북마크 모음
- 엔터프라이즈 eDiscovery, legal hold, 조건별·사용자별 상세 보존 정책
- AI 대화 요약·질의응답
- Slack 화면과 단축키의 완전한 복제

제외 항목은 삭제된 요구사항이 아니다. 실제 고객 흐름과 운영 요구가 확인될 때 별도 milestone로
승격한다.

## 1차 6단계 완성 계획

| 단계 | 범위 | 종료 gate | 상태 |
| --- | --- | --- | --- |
| 1 | Realtime 연결·재접속 안정화 | 전체 wire 리소스 수신, 다중 페이지 복구, 로그아웃 중단·재로그인 재개, 반복 실패 backoff, 변경 burst 묶음 | 완료 |
| 2 | P0 채널 관리·메시지 이동 | 오래된 알림 원문 이동, 멤버 제거, 주제·설명, 보관·복원과 권한 테스트 | 완료 |
| 3 | 알림 완성 | OS 알림, 채널별 수준, DND, 읽음 수 동기화와 플랫폼별 기능 저하 검증 | 완료 |
| 4 | 품질 검증 | 두 사용자 E2E, 오프라인·재연결·토큰 만료, 대화량·반응형·접근성 gate | 완료 |
| 5 | 업무 연결 | 메시지→WorkItem, 원문 역링크, 프로젝트·고객·티켓 문맥 채널 | 완료 |
| 6 | 회의 통합 | 회의 채팅 보존, 녹화·회의록 게시, 안건·후속 작업 전환 | 완료 |

1단계는 실제 개발 환경에서 Keycloak 토큰으로 연결해 과거 변경 92건을 한 연결에서 적용했고,
연결 상태가 `connected`로 유지되는 것을 확인했다. 계약 목록은 Desktop Zod와 서버 JSON Schema가
달라지면 테스트가 실패하며, reconnect catch-up 중 다수 변경은 조직별 한 번의 화면 갱신으로 묶는다.
6단계의 회의 제품 범위와 상세 gate는 [회의·회의록 제품 로드맵](./35-meeting-product-roadmap.md)이 소유한다.

2~6단계도 같은 기준으로 닫았다. 채널 설정과 정확한 메시지 이동, OS 알림·채널별 수준·DND,
50건 초과 페이지 병합과 반응형 화면을 검증했다. 메시지는 idempotent하게 WorkItem으로 전환되고
양쪽에서 원문으로 이동한다. Project·Customer·Ticket·Meeting은 각각 하나의 문맥 채널을 가지며,
회의 상세에서는 그 채널을 열고 확정 회의록과 준비된 녹화를 중복 없이 게시한다. 회의 채널 메시지는
출처 링크를 보존한 안건 또는 기존 WorkItem 전환 흐름을 이용한 후속 작업으로 승격할 수 있다.

## 2026-07-21 안정성·운영 6단계

기능 수를 늘리는 작업 이후에는 실제 사용 중 메시지를 잃지 않고, 긴 대화와 운영 정책까지 감당하는
두 번째 완성 단계를 진행했다.

| 단계 | 범위 | 완료 증거 | 상태 |
|---|---|---|---|
| 1 | 채널·스레드 작성 초안 보존 | 사용자·채널·스레드별 복원, 첨부 ID 보존, 저장소 실패 안전 테스트 | 완료 |
| 2 | 전송 실패와 재시도 | optimistic 실패 표시, 같은 요청 ID 재시도, 타임라인 재시도·취소 | 완료 |
| 3 | 안 읽음과 스크롤 복원 | 서버 read cursor, unread 경계, 새 메시지 점프, 채널·스레드 위치 복원 | 완료 |
| 4 | 대용량 타임라인 | 150건부터 가상화, 가변 높이 측정, 10,000건 병합·DOM 상한 검증 | 완료 |
| 5 | 채널 운영·거버넌스 | 1~3650일 보존, JSON 내보내기, 관리자 삭제 사유, 최근 200건 감사 조회 | 완료 |
| 6 | 통합 검증과 릴리스 점검 | 계약·타입·현지화·DB/API vertical·Electron build gate | 완료 |

보존 실행은 메시지와 수정 이력의 본문을 함께 지우고 tombstone·삭제 행위자·사유는 남긴다. 적용 요청은
`Idempotency-Key`로 중복 실행을 방지한다. 내보내기는 관리자 권한으로 제한되고 최대 10,000건에서
`truncated`를 표시한다. 상세 명령과 검증 범위는
[채팅 릴리스 검증 기록](./37-chat-release-evidence.md)에 유지한다.

## 구현 순서

2026-07-20 현재 C0~C4 Desktop 수직을 구현했다. 최신 페이지와 `before` 과거 커서, realtime
병합, 검색·고정 위치 이동, 작성자 수정·삭제 확인 UI, 첨부만 있는 메시지, 구조화된 그룹 멘션,
멤버 관리 권한이 있는 사용자의 채널 멤버 추가가 연결됐다. Realtime 안정화, 채널 관리, 알림,
업무 문맥과 회의 통합까지 1차 6단계를 닫았고, 기본 채널 retention·내보내기·감사와 대용량
타임라인까지 2차 안정성·운영 단계에서 구현했다. legal hold, eDiscovery와 분산 부하 시험은 별도
엔터프라이즈 운영 milestone로 관리한다.

### C0 · 타임라인 신뢰성

- 채널 진입 시 오래된 첫 페이지가 아니라 최신 메시지를 연다.
- 최신 페이지에서 과거 메시지를 점진적으로 불러온다.
- 새 메시지 수신·재연결·창 focus 후에도 중복과 누락 없이 시간순을 유지한다.
- 검색·고정·알림 결과가 해당 메시지로 이동하고 강조된다.

**종료 gate:** 51개 이상 메시지 채널에서 최신 메시지가 즉시 보이고, 과거 페이지를 불러온 뒤
중복 id가 없으며, 검색 결과 선택 시 목표 메시지에 도달한다.

### C1 · 메시지 소유권 동작

- 작성자는 자기 메시지를 수정한다. 현재 `version`을 `If-Match`로 보내 동시 편집 충돌을 감지한다.
- 작성자는 자기 메시지를 확인 후 삭제한다.
- 다른 사람 메시지에는 수정 메뉴를 노출하지 않는다. `channel.manage` 사용자는 삭제 메뉴를 보고,
  타인 메시지를 삭제할 때 500자 이내 사유를 반드시 남긴다.
- 성공 시 타임라인을 즉시 갱신하고, 실패 시 작성 내용을 보존한 채 오류를 표시한다.

**종료 gate:** 수정 성공·버전 충돌·삭제 취소·삭제 성공·타인 메뉴 미노출을 Renderer 테스트로
검증하고 서버 vertical 테스트를 유지한다.

### C2 · 스레드와 멘션 완결

- 스레드 답글도 사용자 멘션, 리액션, 수정·삭제를 지원한다.
- `@channel`과 `@here`를 구조화된 전송 옵션으로 연결한다.
- 편집 중 삭제된 사용자 멘션이 알림 대상으로 남지 않게 한다.
- 스레드 새 답글을 realtime nudge로 갱신한다.

**종료 gate:** 본문과 스레드에서 같은 메시지 권한·렌더링 규칙이 적용되고, 직접 멘션과 그룹
멘션의 뮤트 관통 규칙이 서버 테스트와 일치한다.

### C3 · 업무 연결

- 메시지 action에서 대상 Team을 선택해 WorkItem을 만든다.
- 생성 중 중복 클릭은 동일 idempotency key로 한 건만 만든다.
- 메시지와 WorkItem 양쪽에서 원문·생성 업무로 이동한다.

**종료 gate:** `message.read`와 `work_item.create`를 모두 가진 사용자만 전환할 수 있고, 같은 요청의
재시도에서 WorkItem과 링크가 중복되지 않는다.

### C4 · 운영 완성

- OS 알림, 알림 설정, DND 시간
- 채널 멤버 관리, 주제·설명, 보관
- retention·내보내기·삭제 정책과 감사 조회
- 대규모 채널의 검색·목록·realtime 부하 시험

## 공통 불변식

- Renderer는 access token을 받거나 저장하지 않는다. Main이 활성 Pie session으로 API를 호출한다.
- 모든 읽기·검색·첨부·알림은 조직 RLS와 채널 멤버십을 우회하지 않는다.
- durable 메시지 변경은 outbox→realtime 경로를 사용하고, typing·presence만 휘발성 경로를 사용한다.
- optimistic UI 실패는 서버 성공으로 위장하지 않는다. 실패 상태 또는 복구 가능한 작성 내용을 남긴다.
- 삭제는 hard delete가 아니라 tombstone이며 감사 메타데이터와 본문 보존 정책을 분리한다.
- 로컬·WSL·SSH host는 채팅 Control Plane 위치를 가정하지 않고 Instance discovery 결과를 사용한다.

## 완료 정의

채팅 코어는 다음을 모두 만족할 때 완료다.

1. 두 사용자로 채널과 DM에서 메시지·스레드·리액션·멘션·첨부를 왕복한다.
2. 50개가 넘는 대화에서도 최신 진입, 과거 불러오기, 실시간 추가가 누락·중복 없이 동작한다.
3. 작성자 수정·삭제, 고정, 검색 이동, 안 읽음·알림·뮤트가 Desktop UI에서 끝까지 연결된다.
4. 비멤버와 교차 조직 사용자는 메시지·검색·첨부·presence를 읽거나 조작할 수 없다.
5. renderer unit, Main IPC/client, Control Plane vertical, 실제 PostgreSQL·Realtime 통합 테스트가 통과한다.
6. macOS·Linux·Windows 패키지 환경에서 입력, 첨부, 링크 열기와 알림의 기능 저하가 명시된다.
