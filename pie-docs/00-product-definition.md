# Pie 제품 정의

## 제품 한 문장

Pie는 프로젝트 업무에서 시작한 사람과 AI의 실행 과정, 코드·문서 산출물, 검토·승인·배포와
고객 증빙까지 연결하는 로컬 우선 SI 프로젝트 포털이자 데스크톱 실행 플랫폼이다.

## 해결하려는 문제

SI 회사의 업무는 CRM, 메신저, 프로젝트 관리, Git, AI 코딩 세션, 원격지원, 문서, 공수 시스템에
흩어진다. 그 결과 고객 요청과 업무의 원문, Claude Code·Codex에서 수행한 판단과 도구 실행,
실제 코드 변경, 검수 결과, 청구 근거가 서로 연결되지 않는다. Pie는 이 정보들을 동일한 업무
문맥으로 연결한다.

## 핵심 사용자

- 경영진과 사업 책임자
- 영업과 고객 담당자
- PM, PL, 품질 담당자
- 개발자와 DevOps 엔지니어
- 서비스 데스크와 유지보수 엔지니어
- 고객사 관리자와 업무 담당자
- 외부 협력사와 감사 담당자

## 단일 앱 원칙

Pie는 역할별로 별도 제품을 만들지 않는다. 같은 Electron 앱이 로그인 계정과 초대 토큰에
따라 다음 모드 중 하나로 동작한다.

| 모드 | 대상 | 제공 범위 |
|---|---|---|
| 운영 모드 | 경영, 영업, PM | 고객, 계약, 프로젝트, 자원, 보고서 |
| 개발 모드 | 개발자, PL | Workspace, Git, 에이전트, 테스트, 배포 |
| 지원 모드 | 서비스 엔지니어 | 티켓, SLA, 자산, 원격지원, 지식베이스 |
| 고객 모드 | 고객사 사용자 | 요청, 진행상태, 채팅, 승인, 검수, 원격지원 |
| 협력사 모드 | 외부 수행 인력 | 배정 업무와 허용된 자료·저장소·세션 |
| 게스트 지원 모드 | 일회성 참가자 | 초대된 원격 세션만 접근 |

고객 모드와 게스트 지원 모드에서도 동일한 설치 파일을 사용한다. 숨겨진 기능은 Renderer뿐
아니라 IPC와 서버 API에서 호출할 수 없어야 한다.

별도 업무용 웹 앱은 만들지 않지만 앱이 없는 사용자의 이메일 확인, 초대, SSO, 설치 복구를 위한
최소 공개 HTTPS 페이지는 제공한다. 이 페이지에서는 고객·프로젝트·티켓 업무를 처리하지 않는다.

## 핵심 가치

### 업무 문맥 연결

고객, 계약, 프로젝트, 요구사항, 티켓, 원격 세션, Worktree, PR, 배포, 검수, 공수를 하나의
관계 그래프로 연결한다.

### AI 실행력

AI는 요약만 하는 보조자가 아니다. 승인된 범위 안에서 로그 수집, 코드 분석, 테스트,
문서 작성, Runbook 실행까지 수행한다.

### 실행 추적성

프로젝트 업무에서 연 Workspace와 Claude Code·Codex 세션을 명시적으로 연결한다. 프롬프트,
응답, 도구 실행, 변경 파일, commit, PR·MR, test와 문서 산출물을 같은 업무 타임라인에서 검토한다.
관찰한 사실과 에이전트의 주장, 경로 기반 추론을 구분해 자동 완료와 승인 근거가 섞이지 않게 한다.

### 증빙 가능한 지원

누가 어떤 장비에 접속했고 무엇을 보거나 실행했는지, 어떤 코드와 배포로 이어졌는지를
감사 가능한 타임라인으로 남긴다.

### 개발 흐름 보존

기존의 병렬 Worktree, 터미널, SSH, 내장 브라우저, Diff 검토 경험을 업무관리 화면 속에
억지로 축소하지 않는다. 프로젝트나 티켓에서 Workspace를 열고 다시 업무 문맥으로 돌아온다.

## 대표 업무 흐름

1. 프로젝트 → WorkItem → Workspace → Claude Code·Codex → 변경·테스트·PR → 검토·완료
2. 고객 요청 → 서비스 티켓 → 채팅·회의 → 원격 진단 → Worktree → 수정·PR → 고객 확인
3. 영업기회 → 견적 → 계약 → 프로젝트 → 인력 배정 → 산출물 → 검수 → 청구
4. 요구사항 → 영향도 분석 → 변경요청 → 비용·일정 승인 → 개발 → 테스트 → 릴리스
5. 모니터링 경고 → 장애 티켓 → 로그 수집 → 복구 Runbook → 원인분석 → 지식문서
6. 유지보수 계약 → SLA·공수 차감 → 월간 보고 → 계약 갱신

## 제품 경계

Pie가 직접 소유하는 영역은 고객·프로젝트·티켓 문맥, 원격지원, 개발 실행, 공수와 증빙이다.
급여, 세무신고, 범용 마케팅 자동화, 범용 사내 메신저 전체를 대체하지 않는다. 필요한 경우
회계, 전자서명, 메일, 일정, 메신저 제품과 연동한다.

## 품질 속성

- macOS, Windows, Linux에서 핵심 업무가 동일해야 한다.
- 네트워크가 불안정해도 작성 중 데이터와 터미널 세션을 최대한 복구해야 한다.
- SSH, WSL, 원격 호스트에서 수행되는 Git과 터미널 작업을 1급 실행 환경으로 취급한다.
- 고객 데이터는 조직과 고객사 경계로 격리한다.
- 모든 권한 상승, 원격 조작, 승인, 내보내기를 감사할 수 있어야 한다.
- 긴 작업과 스트리밍 출력이 Renderer 응답성을 저해하지 않아야 한다.
- 앱과 서버의 버전 차이, 부분 장애, 백업 복원을 정상 운영 시나리오로 검증해야 한다.
- 키보드와 보조기술로 핵심 가입·티켓·원격지원 흐름을 완료할 수 있어야 한다.

## 1차 성공 기준

사용자가 개인 또는 회사 조직에서 프로젝트와 업무를 만들고, 해당 업무에서 Workspace와
Claude Code·Codex를 열어 수정·테스트·PR을 만든 뒤, 세션의 입력·응답·도구 실행과 산출물이
업무 타임라인에 중복 없이 남아야 한다. 네트워크 중단과 앱 재시작 후에도 연결을 복구하고,
내부 사용자와 고객에게 서로 허용된 정보만 보여주며 사람이 결과를 검토해 완료할 수 있어야 한다.
